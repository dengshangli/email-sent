# Gmail SMTP 发信改造设计规格

## 目标

将邮件发送平台的底层投递服务从 Resend 测试域名改为 Gmail SMTP，使平台无需自有域名即可向 Gmail、163、QQ 等有效邮箱地址发送邮件。

平台不再按收件人域名做限制，但仍受 Gmail 的账号额度、反垃圾策略和收件服务器规则约束。个人 Gmail 账号按最多约 500 封或 500 个收件人/天设计；本工具继续定位为本地或内部测试工具，不用于营销群发。

## 发送架构

- 使用 Nodemailer 连接 `smtp.gmail.com`。
- 使用 Gmail 完整邮箱地址和应用专用密码进行服务端认证。
- SMTP 凭据只保存在 `.env.local`，不得返回浏览器、写入日志或提交 Git。
- 发件邮箱固定为认证的 Gmail 账号，可配置中文显示名称，但不允许伪造其他发件地址。
- 复用一个 SMTP transporter，按收件人逐封顺序发送，避免收件人地址互相暴露，也降低短时间并发触发风控的概率。
- React Email 或上传的 HTML 只负责邮件内容；实际投递由 Gmail SMTP 完成。

## 配置

`.env.example` 改为包含：

- `SMTP_USER`：用于发信的完整 Gmail 地址。
- `SMTP_APP_PASSWORD`：开启两步验证后生成的 16 位应用专用密码。
- `EMAIL_FROM_NAME`：发件人显示名称，默认「邮件发送平台」。

启动时不主动联网验证凭据。第一次发送时建立 SMTP 连接；认证失败时返回明确的中文配置提示。README 说明两步验证、应用专用密码、个人 Gmail 额度和重启服务步骤。

## 接口与发送结果

现有 `/api/send` 请求结构保持不变：

- `recipients`：1–50 个已清洗邮箱地址。
- `subject`：非空标题。
- `html`：非空 HTML，最大 1 MB。

服务端继续重复验证所有输入。每个收件人调用一次 `sendMail`，SMTP 信封只包含当前收件人。

全部成功时返回：

- `ok: true`
- `acceptedCount`：Gmail SMTP 已接受的邮件数量。
- `failedRecipients: []`

部分失败时仍继续处理后续收件人，并返回：

- `ok: false`
- `acceptedCount`：已经被 SMTP 接受的数量。
- `failedRecipients`：失败的收件人地址。
- 脱敏中文错误说明。

前端把原来的「已成功发送」改为「邮件服务器已接受」，避免把 SMTP 接受误解为已进入收件箱。部分失败时显示成功数量和失败地址，保留表单内容以便重试。

## 错误处理

- 缺少 SMTP 配置：提示检查 Gmail SMTP 环境变量。
- Gmail 认证失败：提示确认两步验证和应用专用密码，不回显密码或底层认证详情。
- 达到发送额度或触发风控：提示 Gmail 暂停发送，稍后重试。
- 单个地址被拒绝：记录该地址为失败，继续发送其余地址。
- 网络或未知 SMTP 错误：返回统一中文错误，不泄露堆栈、主机响应或凭据。

## 测试与验收

- 单元测试 mock Nodemailer，不连接真实 Gmail。
- 覆盖配置缺失、每位收件人独立信封、全部成功、部分失败、认证失败和错误脱敏。
- 更新 Playwright，验证「邮件服务器已接受」及部分失败提示。
- 运行 lint、单元测试、Playwright 和生产构建。
- 最终使用 Gmail 应用专用密码，分别向账户 Gmail 和一个外部邮箱发送真实测试邮件，并在 Gmail「已发送」与收件邮箱中核对。

## 明确不做

- 不调用 React Email 的公共测试代理接口。
- 不实现 Google OAuth；本地内部工具使用应用专用密码即可。
- 不提供任意 SMTP 服务商切换、账号池、队列、重试任务或群发营销能力。
- 不承诺邮件一定进入主收件箱，也不绕过 Gmail 的额度和反垃圾规则。
